## 1. Основные планы и этапы проекта

### 1.1 Краткое описание проекта

**Название проекта:**  
Sentinel — AI Scheduler (Backend/Серверная часть)

**Цель проекта:**  
Разработать серверную часть для интеллектуального планировщика времени с использованием LLM (Large Language Model) для интерпретации текстовых запросов пользователя и управления событиями календаря через REST API.

**Краткое описание задач:**

- Спроектировать и реализовать REST API с версионированием (/api/v1) и автодокументацией (OpenAPI 3.x)
- Разработать схему базы данных PostgreSQL (5 таблиц: users, chats, chat_messages, events, user_achievements)
- Интегрировать OpenAI SDK для LLM tool-use (function calling) с поддержкой инструментов calendar.search_events и events.create
- Реализовать систему аутентификации и авторизации на основе JWT
- Реализовать управление событиями и напоминаниями с учётом часовых поясов (IANA Time Zone Database)
- Реализовать систему достижений (геймификация)
- Провести модульное и интеграционное тестирование (покрытие ≥70%)
- Подготовить техническую документацию по ГОСТ 19 ЕСПД

### 2.2 Обоснование выбранного технологического стека

**Выбор Python 3.11+ и FastAPI обусловлен:**

1. **Асинхронность:** FastAPI построен на ASGI (Starlette), что обеспечивает высокую производительность при обработке множественных одновременных запросов. Это критично для диалоговых сценариев с LLM, где время ожидания ответа может достигать нескольких секунд.

2. **Интеграция с AI:** Python — стандарт для AI/ML разработки. OpenAI SDK имеет первоклассную поддержку Python с полным набором функций (function calling, streaming, error handling).

3. **Автодокументация:** FastAPI автоматически генерирует OpenAPI 3.x спецификацию, что упрощает интеграцию с фронтендом и соответствует требованиям ТЗ по документированию API.

4. **Типизация и валидация:** Pydantic v2 обеспечивает строгую валидацию входных/выходных данных на уровне схем, что снижает количество runtime-ошибок и улучшает Developer Experience.

5. **Экосистема:** Богатый выбор библиотек для работы с БД (SQLAlchemy), миграциями (Alembic), тестированием (pytest), что ускоряет разработку.

**Выбор PostgreSQL:**

1. **JSONB:** Нативная поддержка JSON-документов для хранения гибких структур (preferences, metadata, tool_args/tool_result в chat_messages).

2. **Расширения:** pg_trgm для быстрого полнотекстового поиска по событиям, pgcrypto для генерации UUID.

3. **Надёжность:** ACID-транзакции, репликация, point-in-time recovery для production-окружения.

4. **Типы данных:** TIMESTAMPTZ для корректной работы с часовыми поясами, TEXT[] для массивов тегов, UUID для идентификаторов.

**Выбор OpenAI SDK:**

1. **Function calling:** Нативная поддержка tool-use для вызова функций поиска событий (calendar.search_events) и создания записей (events.create).

2. **Официальная библиотека:** Стабильность, полная документация, обратная совместимость, регулярные обновления.

3. **Гибкость:** Возможность переключения между моделями (GPT-4, GPT-4 Turbo) без изменения кода, настройка параметров (temperature, max_tokens).

**Преимущества стека для проекта:**

- **Скорость разработки:** FastAPI с автодокументацией и Pydantic-валидацией минимизирует boilerplate-код
- **Типобезопасность:** Статическая типизация (Python + mypy) снижает количество runtime-ошибок
- **Производительность:** Асинхронность (async/await) обеспечивает высокую пропускную способность при работе с LLM и БД
- **Масштабируемость:** Stateless API позволяет горизонтальное масштабирование; PostgreSQL — вертикальное
- **Соответствие требованиям:** Стек полностью покрывает требования ТЗ (REST API, OpenAPI 3.x, JWT, ISO-8601, IANA TZ, JSONB)
- **Экосистема AI:** Python — стандарт для AI/ML, OpenAI SDK имеет первоклассную поддержку

## 4. Особые пометки

### Риски проекта

1. **Недоступность OpenAI API** (вероятность: средняя, влияние: высокое)
   - Меры митигации: реализация graceful degradation, информирование пользователя о временной недоступности, fallback на локальные операции (просмотр существующих событий)

2. **Превышение лимитов OpenAI** (вероятность: низкая, влияние: среднее)
   - Меры митигации: мониторинг использования токенов, rate limiting на уровне приложения, кэширование частых запросов

3. **Сложность интеграции LLM tool-use** (вероятность: средняя, влияние: высокое)
   - Меры митигации: ранний прототип интеграции, изучение документации OpenAI, тестирование различных сценариев диалога, few-shot примеры в system prompt

4. **Проблемы с часовыми поясами** (вероятность: средняя, влияние: среднее)
   - Меры митигации: использование IANA Time Zone Database, тщательное тестирование edge cases, unit-тесты для TZ-логики, хранение всех дат в UTC

5. **Недостаточное покрытие тестами** (вероятность: средняя, влияние: среднее)
   - Меры митигации: выделение времени на написание тестов в плане (этап 2.03-10.03), TDD-подход для критичных компонентов

6. **Отставание от графика** (вероятность: средняя, влияние: высокое)
   - Меры митигации: еженедельные синхронизации с frontend-разработчиком, приоритизация задач, MVP-подход (минимальная жизнеспособная версия)

### Изменения в планах

- **Упрощение модели данных:** Изначально планировалось 8+ таблиц (schedules, blocks, resource_battery, attachments, settings), но для MVP решено использовать минимальную модель из 5 таблиц (users, chats, chat_messages, events, user_achievements). Дополнительные таблицы могут быть добавлены в будущих итерациях.

- **Отказ от мультимодального ввода в MVP:** Функционал распознавания фото/скриншотов через Vision API перенесён на вторую итерацию. В MVP фокус на текстовом диалоге с LLM.

- **Упрощение Resource Battery:** Вместо сложной модели расчёта энергии на backend реализован базовый API для хранения/получения метрик. Основная логика расчёта перенесена на frontend.

### Технические ограничения

- **Отсутствие офлайн-режима:** Backend требует активного подключения к интернету для работы с LLM. При недоступности провайдера функционал создания событий через чат недоступен.

- **Зависимость от внешнего LLM:** Нет собственных ML-моделей, вся интерпретация запросов идёт через OpenAI API. Это упрощает разработку, но создаёт зависимость от внешнего сервиса.

- **Учебный характер проекта:** Проект выполняется в рамках курсового проекта 3 курса бакалавриата. Некоторые production-требования (высокая доступность 99.9%, масштабирование на тысячи пользователей, полный мониторинг и алертинг) упрощены.

### Дополнительные замечания

- **Фокус на backend:** Данный документ описывает исключительно backend-часть проекта, разрабатываемую Максимовым Т.С. Frontend (iOS) разрабатывается Петяновым Т.А. параллельно.

- **Документация по ГОСТ:** Полный комплект документации (ТЗ, ПЗ, ПМИ, РО, ТП) готовится в соответствии с ГОСТ 19 ЕСПД с использованием Typst для вёрстки.

- **Контроль версий:** Весь код хранится в Git-репозитории с осмысленными коммитами, ветвлением по feature-branch модели, code review через pull requests.

- **Синхронизация команды:** Еженедельные встречи для обсуждения прогресса, блокеров, интеграционных вопросов. Коммуникация через Telegram.
